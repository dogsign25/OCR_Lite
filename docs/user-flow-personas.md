## P-01. 대량 처리 담당자 민수

상황:

- 수십 개 이상의 PDF를 한 번에 처리한다.
- 처리 시간이 길어도 전체 작업이 중단되지 않는 것이 중요하다.

흐름:

1. PDF를 `input/`에 넣는다.
2. `python -m app.batch`를 실행한다.
3. `batch_outputs/batch-summary.json`에서 성공, 건너뜀, 실패 건수를 확인한다.
4. 각 PDF 폴더의 `verified.json`을 사용한다.

보호 장치:

- 기본 두 개 PDF 동시 처리
- 한 PDF 실패가 다른 PDF를 중단시키지 않음
- 각 PDF별 독립 출력 폴더
- 완료 manifest가 유효한 작업만 재실행에서 건너뜀

오류 판정:

- 한 PDF가 실패했을 때 다른 PDF도 실패하거나 중단되면 오류
- `batch-summary.json`의 성공·실패 개수가 실제 처리 결과와 다르면 오류
- 실패한 PDF에 완료 manifest가 남으면 오류

## P-02. 문서 갱신 담당자 지영

상황:

- 같은 파일명으로 PDF 내용을 교체한다.
- OCR 언어나 처리 알고리즘이 변경될 수 있다.

흐름:

1. 기존 PDF를 같은 이름의 새 PDF로 교체한다.
2. 배치 명령을 다시 실행한다.
3. 프로그램이 원본 SHA-256과 처리 설정을 비교해 변경된 PDF만 재처리한다.

보호 장치:

- 파일명만으로 완료 여부를 판단하지 않음
- 원본 해시, OCR 언어, 파이프라인 버전, 모든 산출물 존재 여부를 검증
- 결과 파일 하나라도 삭제되면 해당 PDF 자동 재처리

오류 판정:

- 원본 해시나 필터 단어가 바뀌었는데 `skipped`로 처리되면 오류
- manifest에 기록된 산출물이 없는데 이전 결과를 재사용하면 오류

## P-03. 시스템 운영자 현우

상황:

- Tesseract가 없거나 `kor` 언어팩이 설치되지 않았다.
- 일부 PDF가 손상되었거나 특정 페이지 OCR이 실패한다.

흐름:

1. 배치 실행 결과에서 `[FAILED]` 항목을 확인한다.
2. `batch-summary.json`의 `error`를 확인한다.
3. 환경 또는 원본을 수정한 뒤 다시 실행한다.

보호 장치:

- 처리 전에 Tesseract 언어팩을 검증
- 모든 OCR 후보가 실패하면 빈 JSON을 성공 결과로 저장하지 않음
- 실패한 PDF의 부분 PNG와 JSON을 정리
- 다른 PDF 처리는 계속 진행

오류 판정:

- OCR 프로필이 모두 실패했는데 verified JSON을 생성하면 오류
- 일부 프로필 실패가 경고에 기록되지 않으면 오류

## P-04. 품질 검수자 서연

상황:

- 자동 선택 결과를 사용하되 불확실한 페이지만 직접 확인하고 싶다.

흐름:

1. `verified.json`의 `verification.review_required_pages`를 확인한다.
2. `pages[].review_required`가 `true`인 페이지만 PNG와 비교한다.
3. 필요하면 세 candidate JSON을 함께 비교한다.

검수 대상으로 표시되는 조건:

- 텍스트를 찾지 못함
- 선택 결과의 평균 OCR 신뢰도가 60 미만
- 후보 간 평균 일치도가 0.5 미만

주의:

- `verified`는 여러 OCR 결과 중 가장 신뢰할 후보를 자동 선택했다는 뜻이며,
  내용의 절대적 정답을 보증하지 않는다.
- PDF 내장 텍스트는 일부 문서에서 제목만 포함하는 등 불완전할 수 있어
  자동 정답 후보로 사용하지 않는다.

오류 판정:

- 신뢰도 60 미만 페이지가 `review_required`로 표시되지 않으면 오류
- 빈 OCR 후보가 정상 텍스트 후보보다 우선 선택되면 오류

## P-05. 웹 사용자 준호

상황:

- 명령어보다 브라우저 업로드가 편하다.

흐름:

1. 웹 서버를 실행한다.
2. PDF를 업로드한다.
3. verified JSON과 candidate JSON을 다운로드한다.

보호 장치:

- OCR은 작업 스레드에서 실행되어 웹 이벤트 루프를 막지 않음
- 실패 시 해당 업로드의 부분 산출물을 정리
- 검수가 필요한 페이지 수를 결과 카드에 표시

오류 판정:

- 빈 파일 처리 후 업로드·PNG·JSON·필터 PDF가 남으면 오류
- 파일별 실패 이유가 결과 카드에 전달되지 않으면 오류

## P-06. 페이지 필터 사용자 예린

상황:

- 긴 PDF에서 계약 번호, 담당자명, 승인 문구 등이 있는 페이지만 필요하다.
- 원본은 보존하면서 관련 없는 페이지를 제거한 새 PDF를 만들고 싶다.

흐름:

1. 배치 명령에 `--filter-word`를 반복해서 전달하거나 웹 입력란에 단어를
   쉼표 또는 줄바꿈으로 등록한다.
2. 프로그램이 각 페이지의 세 OCR 후보에서 등록 단어를 검색한다.
3. 하나 이상의 단어가 감지된 페이지만 `<이름>.filtered.pdf`에 저장한다.
4. verified JSON의 `page_filter`와 `matched_filter_terms`를 확인한다.

보호 장치:

- 원본 PDF를 수정하지 않음
- 대소문자 및 연속 공백 차이를 무시하고 여러 단어는 OR 조건으로 처리
- 필터 설정이 변경되면 기존 배치 결과를 재사용하지 않음
- 일치 페이지가 없으면 빈 PDF 대신 명확한 실패 결과를 기록

오류 판정:

- 등록 단어가 다른 OCR 후보에 있는데 페이지를 제거하면 오류
- 필터 결과의 페이지 순서 또는 원래 페이지 번호가 바뀌면 오류
- 일치 페이지가 없을 때 원본이 변경되거나 빈 PDF가 생성되면 오류

## P-07. 입력 실수 사용자 다은

상황:

- 파일을 선택하지 않고 실행한다.
- PDF가 아닌 파일을 실수로 업로드한다.
- 내용이 없는 PDF 파일을 업로드한다.

기대 동작:

1. 파일 미선택과 잘못된 확장자는 HTTP 400 오류로 거부한다.
2. 빈 PDF는 해당 파일만 실패 결과로 표시한다.
3. 실패 과정에서 생성된 임시 업로드와 산출물을 모두 정리한다.

오류 판정:

- 잘못된 입력이 OCR 단계까지 전달되면 오류
- 오류 원인을 사용자에게 표시하지 않으면 오류
- 실패 후 부분 파일이 남으면 오류
