## P-01. 대량 처리 담당자 민수

상황:

- 수십 개 이상의 PDF를 한 번에 처리한다.
- 처리 시간이 길어도 전체 작업이 중단되지 않는 것이 중요하다.

흐름:

1. PDF를 `input/`에 넣는다.
2. `python -m app.batch`를 실행한다.
3. `batch_outputs/batch-summary.json`에서 성공, 건너뜀, 실패 건수를 확인한다.
4. 각 PDF 폴더의 `verified.json`을 사용한다.

보호 장치:

- 기본 두 개 PDF 동시 처리
- 한 PDF 실패가 다른 PDF를 중단시키지 않음
- 각 PDF별 독립 출력 폴더
- 완료 manifest가 유효한 작업만 재실행에서 건너뜀

오류 판정:

- 한 PDF가 실패했을 때 다른 PDF도 실패하거나 중단되면 오류
- `batch-summary.json`의 성공·실패 개수가 실제 처리 결과와 다르면 오류
- 실패한 PDF에 완료 manifest가 남으면 오류

## P-02. 문서 갱신 담당자 지영

상황:

- 같은 파일명으로 PDF 내용을 교체한다.
- OCR 언어나 처리 알고리즘이 변경될 수 있다.

흐름:

1. 기존 PDF를 같은 이름의 새 PDF로 교체한다.
2. 배치 명령을 다시 실행한다.
3. 프로그램이 원본 SHA-256과 처리 설정을 비교해 변경된 PDF만 재처리한다.

보호 장치:

- 파일명만으로 완료 여부를 판단하지 않음
- 원본 해시, OCR 언어, 파이프라인 버전, 모든 산출물 존재 여부를 검증
- 결과 파일 하나라도 삭제되면 해당 PDF 자동 재처리
- 재처리가 실패하면 마지막 정상 결과와 manifest 복구
- 손상되거나 내용이 누락된 결과 ZIP은 완료 결과로 재사용하지 않음

오류 판정:

- 원본 해시나 필터 단어가 바뀌었는데 `skipped`로 처리되면 오류
- manifest에 기록된 산출물이 없는데 이전 결과를 재사용하면 오류
- 재처리 실패 후 이전 정상 결과가 삭제되거나 부분 결과로 바뀌면 오류

## P-03. 시스템 운영자 현우

상황:

- Tesseract가 없거나 `kor` 언어팩이 설치되지 않았다.
- 일부 PDF가 손상되었거나 특정 페이지 OCR이 실패한다.

흐름:

1. 배치 실행 결과에서 `[FAILED]` 항목을 확인한다.
2. `batch-summary.json`의 `error`를 확인한다.
3. 환경 또는 원본을 수정한 뒤 다시 실행한다.

보호 장치:

- 처리 전에 Tesseract 언어팩을 검증
- 모든 OCR 후보가 실패하면 빈 JSON을 성공 결과로 저장하지 않음
- 실패한 PDF의 부분 PNG와 JSON을 정리
- 다른 PDF 처리는 계속 진행

오류 판정:

- OCR 프로필이 모두 실패했는데 verified JSON을 생성하면 오류
- 일부 프로필 실패가 경고에 기록되지 않으면 오류

## P-04. 품질 검수자 서연

상황:

- 자동 선택 결과를 사용하되 불확실한 페이지만 직접 확인하고 싶다.

흐름:

1. `verified.json`의 `verification.review_required_pages`를 확인한다.
2. `pages[].review_required`가 `true`인 페이지만 PNG와 비교한다.
3. 필요하면 세 candidate JSON을 함께 비교한다.

검수 대상으로 표시되는 조건:

- 텍스트를 찾지 못함
- 선택 결과의 평균 OCR 신뢰도가 60 미만
- 후보 간 평균 일치도가 0.5 미만

주의:

- `verified`는 여러 OCR 결과 중 가장 신뢰할 후보를 자동 선택했다는 뜻이며,
  내용의 절대적 정답을 보증하지 않는다.
- PDF 내장 텍스트는 일부 문서에서 제목만 포함하는 등 불완전할 수 있어
  자동 정답 후보로 사용하지 않는다.

오류 판정:

- 신뢰도 60 미만 페이지가 `review_required`로 표시되지 않으면 오류
- 빈 OCR 후보가 정상 텍스트 후보보다 우선 선택되면 오류

## P-05. 웹 사용자 준호

상황:

- 명령어보다 브라우저 업로드가 편하다.

흐름:

1. 웹 서버를 실행한다.
2. PDF를 업로드한다.
3. verified JSON과 candidate JSON을 다운로드한다.

보호 장치:

- OCR은 작업 스레드에서 실행되어 웹 이벤트 루프를 막지 않음
- 실패 시 해당 업로드의 부분 산출물을 정리
- 검수가 필요한 페이지 수를 결과 카드에 표시
- 현재 처리 단계, 완료율, 예상 남은 시간 표시
- 검수 필요 페이지만 모아 보는 화면 제공
- 파일 개수, 필터 개수, OCR 수정 크기 제한

오류 판정:

- 빈 파일 처리 후 업로드·PNG·JSON·필터 PDF가 남으면 오류
- 파일별 실패 이유가 결과 카드에 전달되지 않으면 오류
- 모든 파일이 실패했는데 진행 상태가 `completed`면 오류

## P-06. 페이지 필터 사용자 예린

상황:

- 긴 PDF에서 계약 번호, 담당자명, 승인 문구 등이 있는 페이지만 필요하다.
- 원본은 보존하면서 관련 없는 페이지를 제거한 새 PDF를 만들고 싶다.

흐름:

1. 배치 명령에 `--filter-word`, `--filter-mode`, `--exclude-word`를
   전달하거나 웹 입력란에 포함·제외 단어와 OR·AND 조건을 등록한다.
2. 프로그램이 각 페이지의 세 OCR 후보에서 등록 단어를 검색한다.
3. 하나 이상의 단어가 감지된 페이지만 `<이름>.filtered.pdf`에 저장한다.
4. verified JSON의 `page_filter`와 `matched_filter_terms`를 확인한다.

보호 장치:

- 원본 PDF를 수정하지 않음
- 대소문자 및 연속 공백 차이를 무시
- 전각·반각 같은 Unicode 호환 문자 차이를 정규화
- 포함 단어는 OR·AND 중 선택하고 제외 단어는 항상 우선 처리
- 포함 단어 없이 제외 단어만 입력하면 해당 단어가 있는 페이지만 제거
- 필터 설정이 변경되면 기존 배치 결과를 재사용하지 않음
- 일치 페이지가 없으면 빈 PDF 대신 명확한 실패 결과를 기록

오류 판정:

- 등록 단어가 다른 OCR 후보에 있는데 페이지를 제거하면 오류
- AND 조건에서 포함 단어 일부가 누락됐는데 페이지를 보존하면 오류
- 제외 단어가 감지됐는데 포함 조건만 보고 페이지를 보존하면 오류
- 필터 결과의 페이지 순서 또는 원래 페이지 번호가 바뀌면 오류
- 일치 페이지가 없을 때 원본이 변경되거나 빈 PDF가 생성되면 오류

## P-07. 입력 실수 사용자 다은

상황:

- 파일을 선택하지 않고 실행한다.
- PDF가 아닌 파일을 실수로 업로드한다.
- 내용이 없는 PDF 파일을 업로드한다.

기대 동작:

1. 파일 미선택과 잘못된 확장자는 HTTP 400 오류로 거부한다.
2. 빈 PDF는 해당 파일만 실패 결과로 표시한다.
3. 실패 과정에서 생성된 임시 업로드와 산출물을 모두 정리한다.

오류 판정:

- 잘못된 입력이 OCR 단계까지 전달되면 오류
- 오류 원인을 사용자에게 표시하지 않으면 오류
- 실패 후 부분 파일이 남으면 오류

## P-08. OCR 교정 담당자 유진

상황:

- 자동 OCR 결과의 오탈자를 직접 수정한다.
- 수정한 문장을 PDF에서도 검색하고 싶다.

흐름:

1. 웹 결과에서 `Edit OCR text`를 펼친다.
2. 페이지별 텍스트를 수정하고 저장한다.
3. verified JSON과 검색 가능한 PDF를 다시 내려받는다.

보호 장치:

- 존재하지 않는 페이지 번호의 수정 요청 거부
- 최초 OCR 텍스트와 수정 시각 기록
- 수정 완료 페이지의 검수 필요 상태 해제
- 수정된 텍스트로 searchable PDF 즉시 재생성
- candidate JSON과 원본 PDF는 변경하지 않음

오류 판정:

- verified JSON과 searchable PDF의 텍스트가 다르면 오류
- 수정 전 OCR 텍스트가 보존되지 않으면 오류
- 잘못된 페이지 번호가 저장되면 오류

## P-09. 결과 전달 담당자 하린

상황:

- OCR 결과를 다른 사람에게 전달할 때 관련 파일을 빠뜨리지 않아야 한다.

기대 동작:

1. 웹 또는 배치 결과에서 `<이름>.results.zip`을 받는다.
2. ZIP의 `images/`, `json/`, `pdf/` 폴더에서 산출물을 확인한다.
3. OCR 수정이나 페이지 편집 후 ZIP을 다시 받으면 최신 결과가 포함된다.

오류 판정:

- 검색 PDF나 verified JSON이 ZIP에서 누락되면 오류
- 수정 전 JSON 또는 PDF가 ZIP에 남으면 오류
- 손상된 ZIP을 정상 배치 결과로 재사용하면 오류

## P-10. 문서 편집 담당자 태호

상황:

- 스캔 방향이 잘못되었거나 문서 페이지 순서를 바꿔야 한다.

기대 동작:

1. 검수 화면에서 페이지를 이동하고 90도 단위로 회전한다.
2. 편집 PDF와 검색 PDF를 다시 생성한다.
3. JSON의 원본 `page_number`와 출력 `output_page_number`를 확인한다.

오류 판정:

- 같은 페이지가 중복되거나 일부 페이지가 사라지면 오류
- 편집 PDF와 검색 PDF의 순서·회전이 다르면 오류
- OCR 수정 후 페이지 배치가 원래 상태로 돌아가면 오류

## P-11. 장시간 작업 사용자 수진

상황:

- 페이지가 많은 PDF를 처리하면서 작업이 멈췄는지 판단하고 싶다.

기대 동작:

1. 업로드 후 OCR 페이지와 프로필 진행 상황을 확인한다.
2. 완료율과 예상 남은 시간이 계속 갱신되는지 확인한다.
3. 성공 또는 일부 실패 상태로 100%에서 종료되는지 확인한다.

오류 판정:

- 진행률이 뒤로 가거나 100%를 초과하면 오류
- 완료된 작업의 남은 시간이 0이 아니면 오류

## P-12. 동시 검수 담당자 은지

상황:

- 같은 문서를 두 브라우저 탭에서 열어 OCR 수정과 페이지 편집을 거의 동시에
  실행할 수 있다.

보호 장치:

- 문서별 잠금으로 OCR 수정과 페이지 편집을 순서대로 처리
- JSON, 검색 PDF, 편집 PDF를 임시 파일에 모두 생성한 뒤 일괄 교체
- ZIP 재생성까지 실패하면 수정 전 산출물로 전체 롤백
- 중복 페이지 번호와 후보 JSON 일부 누락 상태를 거부

오류 판정:

- verified JSON과 검색 PDF가 서로 다른 요청의 상태를 가지면 오류
- 파일 생성 중 장애 후 일부 산출물만 새 상태로 남으면 오류
- 후보 JSON 일부만 순서가 변경되면 오류

## P-13. 비정상 입력 대응 운영자 정우

상황:

- 지나치게 많은 파일, 수백 페이지를 넘는 PDF, 비정상적으로 큰 페이지,
  조작된 ZIP 경로가 시스템 자원을 고갈시키거나 파일 경계를 벗어나면 안 된다.

보호 장치:

- 웹 요청당 PDF 최대 20개
- PDF당 최대 500페이지
- 페이지 렌더 결과 최대 2,500만 픽셀
- 포함·제외 필터 합계 최대 100개
- OCR 수정 텍스트 합계 최대 500만 자
- ZIP 내부 절대 경로와 `..` 경로 거부

오류 판정:

- 제한을 넘긴 PDF를 렌더하거나 부분 PNG를 남기면 오류
- ZIP 항목이 지정 폴더 밖으로 추출될 수 있으면 오류
- 중복 작업 ID가 기존 진행 상태를 덮어쓰면 오류

## P-14. 재처리 담당자 소라

상황:

- 원본 변경 또는 `--overwrite`로 기존 문서를 다시 처리한다.
- 새 OCR이 실패해도 마지막으로 성공한 결과는 계속 사용할 수 있어야 한다.

보호 장치:

- 기존 배치 산출물의 스냅샷 생성
- 새 처리와 manifest 기록이 모두 성공한 뒤 결과 확정
- 실패 시 부분 산출물을 제거하고 이전 파일 복구
- 재귀 검색 시 출력 폴더 하위 PDF 제외
- 대소문자만 다른 입력 파일도 서로 다른 출력 이름 사용

오류 판정:

- 실패한 재처리가 기존 PNG, JSON, PDF, ZIP, manifest를 삭제하면 오류
- 출력 폴더의 검색 PDF를 다음 배치 입력으로 다시 처리하면 오류
- 대소문자만 다른 파일이 같은 결과 폴더를 공유하면 오류
